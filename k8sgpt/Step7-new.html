kubectl-ai로 대화하듯 관리하세요

강력한 오픈소스와 Google의 AI를 결합하여 Kubernetes 클러스터 관리를 혁신하세요.

kubectl-ai 상세 가이드 - AI 기반 Kubernetes 관리: 설치부터 고급 활용까지 -

개요: 오픈소스 AI 어시스턴트, kubectl-ai

공식 Google Cloud Platform github에 공개된 오픈소스 프로젝트 `kubectl-ai`는 자연어 처리(NLP) 기능을 `kubectl`에 통합하여, 사용자가 일상적인 언어로 Kubernetes 클러스터와 상호작용할 수 있게 해주는 강력한 도구입니다. 복잡한 명령어를 기억할 필요 없이, 원하는 작업을 설명하면 AI(Gemini 모델)가 적절한 명령어로 변환해 줍니다.

kubectl-ai 작동 원리 플로우

사용자의 자연어 입력이 Gemini API를 통해 kubectl 명령어로 변환되어 클러스터에 적용되는 과정을 시각화한 다이어그램입니다.

개념 이해: kubectl-ai의 작동 방식

`kubectl-ai`를 더 깊이 있게 활용하기 위해 핵심 개념인 토큰, 가격 정책, 모델 버전에 대해 알아보겠습니다.

토큰(Token)이란 무엇인가요?

토큰은 AI 모델이 텍스트를 처리하는 기본 단위입니다. API 사용 요금은 바로 이 '토큰'의 수를 기준으로 책정됩니다. 일반적으로 단어 하나가 1~2개의 토큰에 해당하지만, 한글과 같은 언어에서는 글자나 형태소 단위로 더 잘게 나뉘기도 합니다.

예시) "쿠버네티스 파드 로그 보여줘" 라는 문장은 다음과 같이 토큰으로 분리될 수 있습니다:
["쿠버네티스", " ", "파드", " ", "로그", " ", "보여줘"]

이 경우 약 7개의 토큰이 사용된 것으로 계산될 수 있습니다. 이처럼 입력하는 프롬프트와 AI가 생성하는 답변의 길이에 따라 사용되는 토큰의 양이 결정됩니다.

Gemini Flash API 가격 정책

대부분의 텍스트 기반 명령어 변환 작업은 매우 저렴한 비용으로 수행되며, 무료 등급으로도 충분한 테스트가 가능합니다.

용어 설명

  • GA (General Availability): '정식 출시'를 의미하며, 모든 사용자에게 안정적으로 제공되는 버전입니다.
  • Google 검색 그라운딩 (Grounding): AI가 답변을 생성할 때 Google 검색 결과를 참고하여 더 정확하고 최신 정보를 반영하는 기능입니다.

3단계 빠른 시작 가이드

단 3단계만으로 `kubectl-ai`를 사용하여 AI 기반 클러스터 관리를 즉시 시작할 수 있습니다.

1단계: Gemini API 키 발급

`kubectl-ai`의 두뇌 역할을 하는 Gemini 모델에 접근하기 위해 API 키가 필요합니다. Google AI Studio에서 무료로 발급받을 수 있습니다.

2단계: kubectl-ai 설치

터미널에 아래 명령어를 붙여넣어 설치 스크립트를 실행합니다. 스크립트가 자동으로 사용자의 시스템 환경에 맞는 `kubectl-ai`를 설치합니다.

Installation Command
curl -sSL https://raw.githubusercontent.com/GoogleCloudPlatform/kubectl-ai/main/install.sh | bash

3단계: API 키 환경 변수 설정

발급받은 API 키를 환경 변수로 설정하여 `kubectl-ai`가 API를 호출할 수 있도록 합니다. 영구적인 사용을 위해 셸 프로필(`.zshrc`, `.bashrc` 등)에 추가하는 것을 권장합니다.

Environment Variable Setup
export GEMINI_API_KEY="[여기에 발급받은 API 키를 입력하세요]"

오픈소스 kubectl-ai 실전 활용 예시 (EKS 환경)

`kubectl-ai`는 단순히 명령어를 한번 실행하는 것 외에도, AI와 대화하며 문제를 해결해 나가는 **대화형 모드**를 지원합니다. 아래 예시들은 실제 **EKS 운영 환경**에서 마주할 수 있는 복잡한 문제들을 어떻게 해결하는지 보여줍니다.

창의적 문제 해결 시나리오 (실행 결과 포함)

1. 목표: EKS 클러스터 전체의 비정상 파드 즉시 확인

'Running' 또는 'Completed' 상태가 아닌 모든 파드를 찾아 즉각적인 조치가 필요한 대상을 식별합니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "모든 네임스페이스에서 'Running'이나 'Completed' 상태가 아닌 모든 파드를 찾아줘"
Execution Result
Running: kubectl get pods --all-namespaces -o jsonpath='{range .items[*]}{.metadata.namespace}{"\t"}{.metadata.name}{"\t"}{.status.phase}{"\n"}{end}' | grep -v "Running" | grep -v "Completed"

'Running' 또는 'Completed' 상태가 아닌 모든 네임스페이스의 파드는 다음과 같습니다:
• 네임스페이스: default, 파드: pending-pod, 상태: Pending
2. 목표: 리소스 요청량이 없는 파드 식별 (비용 최적화)

리소스 요청(requests)이 누락되어 클러스터 스케줄링 효율을 저해하는 파드를 찾아냅니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "모든 파드를 검사해서 CPU나 메모리 request가 설정되지 않은 파드들을 알려줘"
Execution Result
Running: kubectl get pods --all-namespaces -o json | jq -c '.items[] | {name: .metadata.name, namespace: .metadata.namespace, containers: [.spec.containers[] | {name: .name, cpu_request: (.resources.requests.cpu // "not_set"), memory_request: (.resources.requests.memory // "not_set")}]}' | grep -e '"cpu_request":"not_set"' -e '"memory_request":"not_set"'

CPU 또는 메모리 요청이 설정되지 않은 파드들은 다음과 같습니다:
• 네임스페이스: default, 파드: nginx-deployment-5654578ff8-ftwbr
  • 컨테이너: nginx (CPU 요청 없음, 메모리 요청 없음)
• 네임스페이스: default, 파드: pending-pod
  • 컨테이너: pending-pod (CPU 요청 없음, 메모리 요청 없음)
3. 목표: EKS 클러스터 내 미사용 ConfigMap 정리

어떤 파드나 디플로이먼트도 참조하지 않는 ConfigMap을 찾아 불필요한 설정을 정리합니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "어떤 파드도 사용하지 않는 모든 ConfigMap을 찾아줘"
Execution Result
Running: kubectl get configmaps -o json
Running: kubectl get pods -A -o json

분석 결과, 현재 클러스터에는 사용되지 않는 ConfigMap이 없습니다. 모든 ConfigMap은 하나 이상의 파드에서 사용 중입니다.
4. 목표: 모든 네임스페이스의 리소스 사용량 종합 리포트

클러스터 전체의 리소스 사용 현황을 파악하여 용량 계획을 수립합니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "모든 네임스페이스 별로 파드들의 총 CPU 및 메모리 사용량을 요약해서 보여줘"
Execution Result
Running: kubectl top pods --all-namespaces

네, 각 네임스페이스별 파드들의 총 CPU 및 메모리 사용량 요약입니다:

네임스페이스      총 CPU 사용량 (밀리코어)   총 메모리 사용량 (MiB)
--------------------------------------------------------------
default           0m                       4Mi
ingress-nginx     1m                       43Mi
kube-system       25m                      284Mi
kubecost          17m                      1761Mi
5. 목표: EKS 클러스터의 모든 비정상 이벤트 요약

클러스터 전반의 경고 및 에러 이벤트를 확인하여 잠재적인 문제의 징후를 빠르게 파악합니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "지난 1시간 동안 발생한 모든 경고(Warning) 이벤트를 요약해서 보여줘"
Execution Result
지난 1시간 동안 발생한 경고(Warning) 이벤트는 다음과 같습니다:

• 네임스페이스: default
  • 이름: karpenter-test-app-hpa.1847ef93f1c2aba4
  • 마지막 발생 시간: 2025-06-14T00:56:52Z
  • 이유: FailedGetScale
  • 메시지: deployments/scale.apps "karpenter-test-app" not found

• 네임스페이스: default
  • 이름: pending-pod.1848c25a2a5f3a5c
  • 마지막 발생 시간: 2025-06-14T00:59:27Z
  • 이유: FailedScheduling
  • 메시지: 0/1 nodes are available: 1 node(s) didn't match Pod's node affinity/selector.
6. 목표: 특정 파드의 최신 로그 확인

가장 일반적인 디버깅 작업으로, 특정 파드의 최신 로그를 확인하여 현재 상태를 모니터링합니다.

Prompt
kubectl-ai --quiet --model gemini-2.0-flash "kube-system 네임스페이스에 있는 aws-node 파드 중 하나의 최근 로그 10줄을 보여줘"
Execution Result
Running: kubectl logs -n kube-system -l k8s-app=aws-node --tail=10

time="2025-06-14T01:10:00Z" level=info msg="Starting IPamd"
time="2025-06-14T01:10:00Z" level=info msg="CNI-plugin version is: v1.12.6"

결론: 오픈소스와 AI가 만드는 새로운 차원의 개발 경험

오픈소스 도구인 `kubectl-ai`는 Google의 강력한 AI 모델과 결합하여 Kubernetes와의 상호작용 방식을 근본적으로 바꾸는 잠재력을 가지고 있습니다. 복잡한 시스템을 더 인간적인 방식으로 이해하고 제어할 수 있게 함으로써, 클라우드 네이티브 환경의 진입 장벽을 낮추고 전문가의 생산성은 극대화합니다. 이는 '사람이 느낄 수 있는 기술'의 좋은 예시입니다.
